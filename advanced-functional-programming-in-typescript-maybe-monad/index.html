Introducing Maybe

One way of simplifying code is to identify a pattern and create an abstraction that hides the implementation details of this pattern. The recurring theme in the getSupervisorName function is the nesting of if statements.

But how to create an abstraction over such a pattern? The reason we have to do these if checks is that the value stored inside result can be empty. We‚Äôll create a simple wrapper type that holds a simple value and is aware of whether the value is empty (ie. null or undefined or empty string) or not. Let‚Äôs call this wrapper type Maybe.

Instances of Maybe hold a value that can either be an actual value or null. Here, null is the internal representation of an empty value. The constructor is private so you can only create Maybe instances by calling some or none static methods. fromValue is a convenience method that transforms a regular value to a Maybe instance. Finally, getOrElse is a safe way of extracting the value contained by Maybe. The caller has to provide the default value that will be used in case Maybe is empty. So far, so good. Maybe Monad in action

Now, we‚Äôve got all we need to rewrite the getSupervisorName function.

We‚Äôve eliminated all of the nested if statements! The getSupervisorName function‚Äôs body is now an elegant pipeline of transformations applied to the input value. We‚Äôve hidden the details of handling empty results because they‚Äôre just boilerplate and obfuscate the real intention of the code. They‚Äôre now taken care of by Maybe. Note that if any of the operations inside flatMap returned a none, it would cause the whole thing to immediately return a none. This is actually the same behaviour that we had with nested if statements. Here is an example of how the function can be used inside the click handler:

And, guess what, Maybe is a monad! The formal definition of a monad is that it‚Äôs a container type that has two operations:

  • return - which creates an instance of the type from a regular value (some and none in our case)
  • bind - which lets you combine monadic values (flatMap in our case)

There are also some monadic laws that every monad has to follow but let‚Äôs not dive into it yet. For now, you have to trust me that our Maybe implementation follows these laws :)

In this first post of the series, we‚Äôve created our first monad. The purpose of the Maybe monad is to abstract away handling of empty values. Thanks to the introduction of this type, we can now write code without having to worry about empty results.
